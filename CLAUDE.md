### Status surfacing + score history (Bundle 2 + 3, 2026-05-24)

- Status is **event-sourced from Risk_Updates**, not maintained on Risk_Register. The latest non-null `Risk_Updates[updated_status]` per risk (newest-first by update_date+update_id) defines current status via `[Current Status]` measure. `Risk_Register[status]` stays in the schema (hidden) for backwards compat but is not read.
- Backfill rule: terminal statuses (Closed/Realized) appear ONLY on the chronologically-last update for each risk. Earlier updates default to Open or Monitoring even if their notes mention closure language.
- Score history B2 (per Phase 2 §c-Q4 update agreed with user, supersedes original "no score history" decision): 3 nullable int64 columns on Risk_Updates — `probability_at_update`, `cost_impact_at_update`, `schedule_impact_at_update`. Going forward, `append_updates.py` regex populates from mitigation_log patterns.
- Score lifecycle (DAX `[Score At Update]` and `[Score At Update Next]`):
  - Latest status = "Closed" → score returns 0.
  - Latest status = "Realized" → score frozen at the first-Realized update's value (uses `ScoreDate = MIN(update_date where updated_status = "Realized")` for lookback evaluation).
  - Otherwise → lookback formula: for each P/C/S component, find the latest non-null *_at_update value on or before this date, COALESCE to Risk_Register baseline. Step-and-hold semantics: once set, persists until next change.
- Risk_Updates column order frozen (append_updates.py depends on it): `update_id, risk_id, update_date, update_year, author, note, updated_status, probability_at_update, cost_impact_at_update, schedule_impact_at_update`.
